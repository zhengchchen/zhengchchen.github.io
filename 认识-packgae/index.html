认识 packgae

2023.12.03

npm 是 node.js 原生自带的包管理工具（package manager），而 npmjs.org 是 node.js 官方包仓库，也被称为 npm registry。除了 npm 之外，还有诸多包管理工具，比如 yarn、pnpm 等。

npm 与 package

Package

而 package 指的是拥有一个 package.json 的一个文件夹（或压缩包），package 的属性就是 package.json 中文件的内容，例如：

  • name：包名，唯一
  • version：版本号
  • main：默认引入的文件
  • homepage：包的官网或者文档

因为包的名称是唯一的，我们可以通过包名定位到 package 和对应的文档。

在 npmjs.com 中，比如 https://www.npmjs.com/package/<package-name>，只要知道了该包的名字，就可以快速定位到该包的 npm 仓库页面，并了解到关于该包的诸多信息。例如：https://www.npmjs.com/package/lodash

基于以上原理，可以通过 npm docs 命令快速找到一个 package 的官方文档。

npm docs lodash

真实的 package

package 指拥有 package.json 的一个文件夹，但是我们所使用的 package 往往不是我们原地啊吗目录的文件夹，因为包都是经过构建的。一个 package 的真实面目可以在 node_modules 下找到。或者，在某些 cdn 上根据某个 package 的 name 进行快速定位，比如：

  • https://unpkg.com/lodash/
  • https://cdn.jsdelivr.net/npm/lodash/

semver

关于 package 最重要且必须的属性是 name 与 version。semver，即 semantic versioning，语义话文本，它由[major, minor, patch]三部分组成，例如 17.1.1，其中

  • major：当发布一个 breaking change 的 api 时，递增 major 版本号，即 18.0.0
  • minor：当新增了一个向后兼容的功能后，递增 minor 版本号，即 17.2.0
  • patch：当修复了一个向后兼容的 bug 时，递增 patch 版本号，即 17.1.2

在一些大型包管理中，如果没有正式发布，会选择使用 prerelease 的版本号，例如 1.0.0-alpha。

版本号范围

当我们手动安装一个包，它写在 package.json 中的即为版本号范围。

{
  dependencies: {
    midash: "~1.8.1";
  }
}

对于 ～ 1.2.3 而言，它的版本号范围是 >= 1.2.3 且 < 1.3.0

对于 ^1.2.3 而言，它的版本号范围是 >= 1.2.3 且 < 2.0.0

当我们使用 npm i 时，默认的版本号为 ^，可最大限度地在向后兼容与新特性之间做取舍，但是有些库可能不遵循改规则，我们在项目应当使用 yarn.lack 或 package-lock.json 来锁定版本号。

版本号计算

可以通过semver 包进行版本号的计算，例如

semver.valid("1.2.3"); // '1.2.3'
semver.valid("a.b.c"); // null
semver.clean("  =v1.2.3   "); // '1.2.3'
semver.satisfies("1.2.3", "1.x || >=2.5.0 || 5.0.0 - 7.2.3"); // true
semver.gt("1.2.3", "9.8.7"); // false
semver.lt("1.2.3", "9.8.7"); // true
semver.minVersion(">=1.0.0"); // '1.0.0'
semver.valid(semver.coerce("v2")); // '2.0.0'
semver.valid(semver.coerce("42.6.7.9.3-alpha")); // '42.6.7'

依赖

npm i <package>

通过 npm i 进行下载，会下载最新版本的包，写入 package.json 文件中的dependencies字段

{
  "dependencies": {
    "lodash": "^4.17.21"
  }
}

如果通过 npm i -D lodash，将会写入 package.json 文件中的devDependencies字段中。

URI as dependency

依赖页可以通过 URI 的方式安装，例如

$ npm i git+ssh://git@github.com:npm/cli.git

也可通过压缩包及本地文件的形式进行安装。

{
  "dependencies": {
    "npm": "git+ssh://git@github.com:npm/cli.git",
    "foo": "http://q.shanyue.tech/foo.tar.gz",
    "bar": "file:../bar"
  }
}

alias

如果需要在项目中使用某个包的不同版本，可以通过别名的方式引入。

$ npm install <alias>@npm:<name>

$ npm install vue2@npm:vue@2
$ npm install vue3@npm:vue@3

在package中依赖如下：

{
  "dependencies": {
    "vue2": "npm:vue@2",
    "vue3": "npm:vue@3"
  }
}

开发依赖

包开发区别

  • dependencies: 在生产环节使用

  • devDependencies: 在开发环境使用，webpack/babel/eslint 等

  • 只有 dependencies 中的被会被包的依赖安装，zero dependencies 指的是不依赖任何 dependencies

业务开发

当进行业务开发时，二者无严格区别。一般需要加上 private 字段。

install size

指的是某个包的所有依赖及间接依赖（递归 pkg 的所有 dependencies 的总和），可以在https://pkg-size.dev/查看某个包的安装体积。

engines

制定某个项目所需的node最小版本

{
  "engines": {
    "node": ">=14.0.0"
  }
}
  • 如果版本不匹配会报错（yarn）或者警告（npm）
  • 最为重要的是，项目中某些依赖所需要的 Node 版本号与项目运行时的 Node 版本号不匹配，也会报错(在 yarn 中)

nodejs LTS

nodejs 的版本控制采用了semver，即语义话控制。对于 nodejs 这类大型开源项目来说，偶数的版本代表稳定版本，奇数的版本代表实验版本，对于 nodejs 主要由以下两类版本：

  • LTS：Long Term Support，长期支持版本。一般会两年发布一个 LTS，并维护 30 个月的时间，公司生产环境一般应选 LTS。
  • Current：用以展示最新的功能和实验特性，因为 api 可能会经常变化，不推荐使用。 因此对开发 npm 包来说，只需要支持到上一个 LTS 版本即可。

npm script

npm script 是在 javascript 项目中定义脚本的方法。可以通过 npm run <command> 的方式快速执行脚本。允许通过运行一些命令来自动化大部分常见任务，例如编译代码、运行测试、启动服务器等。它可以让开发者快速开发、构建和部署项目，也使得项目的配置变得更加简单和易于维护。 通过 package.json 中的 script 字段进行配置：

{
  "scripts": {
    "dev": "next",
    "start": "next start"
  }
}

在 package.json 默认的 scripts 有：

  • install：依赖安装
  • start：启动服务
  • test：测试项目 他们可以通过 npm <command> 直接运行，如 npm install、npm start。 除了默认的 scripts 外，还有一些约定俗成的脚本，比如：
  • build：构建打包
  • dev：开发环境
  • lint：格式化

对于此类自定义的 scripts 需要 npm run <command> 方可执行。

scirpt 定义的实际上是命令行命令。

如环境变量，NODE_ENV=production 是 Linux 中常见的配置环境变量的方式，在执行命令时读取环境变量 NODE_ENV 为 production。

{
  "scripts": {
    "dev": "NODE_ENV=production webpack",

    "dev:cross-platform": "cross-env NODE_ENV=production webpack"
  }
}

pre/post script

可以在 npm script 中用一些钩子进行辅助开发，提升效率。比如以下问题：

  1. 在 npm publish 发包之前忘记 npm run build，导致发包无效
  2. 在 npm start 启动服务之前需要执行某命令以进行配置数据，但每次都需要记得执行命令，很是繁琐
  3. npm scripts 有两个最为重要的钩子，即 pre/post，这些钩子允许我们在 npm script 执行前后自动执行自定义脚本，可以用于执行一些预处理或后处理任务。
  4. 当我们在手动执行 npm run xxx 时，若果 prexxx 及 postxxx 在 scripts 存在时，它将会自动执行 npm run prexxx 以及 npm run postxxx。
{
  "scripts": {
    "prelint": "echo 'Preparing to lint...'",
    "lint": "eslint",
    "postlint": "echo 'liting complete.'"
  }
}

tips:pnpm 中默认未开启 pre/post 钩子，如果需要使用则需要配置 enable-pre-post-scripts

lifecycle script

除了简单的 pre/post 外，npm publish还有更复杂的生命周期，大概分为以下三阶段：

  1. npm run prepare
  2. npm pack
  3. npm publish

prepare

prepare 将自动发生在以下两个阶段：

  • npm install 之后自动执行，因此有时 prepare 会代替 postinstall 的功能
  • npm publish 之前自动执行

pack

npm pack 将当前 npm 包的内容进行压缩打包为 tarball，这也是实际上传到 npm 仓库的内容。 在我们 npm publish 发包之前，会自动执行 npm pack 压缩打包。在 npm pack 时，会自动计算出 integrity 等信息，方便 npm i 时进行完整性校验。

publish

npm publish 将本地的 npm 包进行 npm pack 打包，并上传到 npm 仓库。以下就是 publish 相关的钩子，其中最为重要的是 prepublishOnly 与 prepare。见 npm publish

  • prepublishOnly
  • prepack
  • prepare
  • postpack
  • publish
  • postpublish
